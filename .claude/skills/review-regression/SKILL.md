# Review Regression — デグレチェック

変更が既存機能を壊していないかを確認する。

## 対象の特定

1. 変更ファイルと差分を取得する:

```sh
git diff main --name-only
git diff main
```

2. 変更ファイルの全体を読む
3. 変更ファイルを参照・利用している関連ファイルを特定して読む。特に:
   - 変更した関数・API を呼び出しているファイル
   - 変更した CSS クラスを使用しているファイル
   - 変更したストレージキーを読み書きしているファイル

## 確認観点

### API 互換性

- `Toolkit.*` の公開 API（引数・戻り値・挙動）が変わっていないか
- 変更した関数のシグネチャが呼び出し元と一致しているか
- 削除・リネームしたものが他のファイルで参照されていないか

### データ互換性

- `chrome.storage` のキー名・データ構造が変わっていないか（既存の保存データが読めなくなる）
- `TAB_MANIFEST` の `storageKey` と実際の使用箇所が一致しているか
- `bindState` のフィールドマップが既存のストレージ構造と整合しているか

### DOM / CSS の影響

- 変更・削除した CSS クラス名が他の HTML / JS で使われていないか
- HTML 構造の変更が CSS セレクタ（子孫・隣接セレクタ等）を壊していないか
- `id` の変更が `querySelector('#...')` や `getElementById` に影響しないか

### ロジックの影響

- 条件分岐の変更がエッジケースの挙動を変えていないか
- イベントハンドラの登録・解除の変更が意図しない多重登録や未登録を起こさないか
- 非同期処理の順序や `loadState` コールバックの前提が崩れていないか

### ドキュメント整合性

- ツールの挙動を変更した場合、README の該当セクションが実装と一致しているか
- 新規ツール追加時に README の機能一覧にセクションが追加されているか

## 出力

指摘事項を以下の形式で報告する:

```
### デグレチェック結果

- [ ] `ファイル名:行番号` — 指摘内容（影響を受ける箇所: `関連ファイル:行番号`）

指摘なしの場合: 「デグレの懸念は見つかりませんでした」
```

修正が求められている場合（issue-handler 経由）は、報告後に指摘箇所を修正する。
