# Review UX — UX レビュー

変更が利用者にとって使いやすいかを確認し、改善点を指摘する。

## 対象の特定

1. 変更ファイルを取得する:

```sh
git diff main --name-only
```

2. 変更ファイルの全体を読む（UI コードは diff だけでは全体像が掴めない）
3. 関連する HTML / CSS / JS を読み、画面上での見え方・動き方を把握する

## 確認観点

### 操作フィードバック

- ボタンクリックやフォーム送信後に、結果が視覚的に伝わるか（トースト、状態変化、ハイライト等）
- 処理に時間がかかる場合、ローディング状態が表示されるか
- 成功・失敗が明確に区別できるか

### エラーハンドリング

- 不正な入力に対してわかりやすいエラーメッセージが出るか
- エラー状態からの復帰手段があるか（入力をクリアする、再試行するなど）
- 空の入力や境界値（0、上限値）で壊れないか

### キーボード操作

- フォーカス可能な要素に `tabindex` が適切に設定されているか
- Enter キーでのフォーム送信・ボタン押下が動作するか
- フォーカス順序が視覚的な配置と一致しているか

### 視覚的一貫性

- 既存のツールと見た目・操作感が揃っているか（ボタンサイズ、余白、フォント）
- 既存の CSS クラス（`.tm-btn`, `.tm-input`, `.tm-textarea` 等）を使っているか
- アイコンのスタイル（`ICONS` 定数の SVG）が他のタブと統一されているか

### アクセシビリティ

- ボタンやアイコンに `title` や `aria-label` があるか
- 色だけに頼らない情報伝達になっているか（色覚多様性への配慮）
- フォーカス時の視覚的インジケータ（アウトライン等）があるか

## 出力

指摘事項を以下の形式で報告する:

```
### UX レビュー結果

**操作フィードバック**
- [ ] `ファイル名:行番号` — 指摘内容（改善案）

**エラーハンドリング**
- [ ] ...

**キーボード操作**
- [ ] ...

**視覚的一貫性**
- [ ] ...

**アクセシビリティ**
- [ ] ...

該当カテゴリに指摘がなければそのカテゴリごと省略する。
全て指摘なしの場合: 「UX 上の問題は見つかりませんでした」
```

修正が求められている場合（issue-handler 経由）は、報告後に指摘箇所を修正する。
