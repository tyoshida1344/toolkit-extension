# Release Publisher — リリース公開スキル

「変更を集める → 対話でリリースノートを固める → バージョンを決める → **ユーザ承認を得てから公開する**」の流れで
GitHub Release を公開する。このスキルの肝は 2 つ:

1. **本文はユーザとの対話で確定する。** コミット履歴はあくまで叩き台の材料であり、勝手に確定して公開しない。
2. **公開は外部への不可逆的な操作。** タグ付け・`gh release create` の実行前に、必ず内容をまとめて提示し明示的な承認を得る。

書式・バージョニング・タグ命名などの規約は **`RELEASING.md` を唯一の出典とする**（ここに重複して書かず、必ず読んで従う）。

## 入力

公開したいバージョン（例: `release-publisher 1.1.0`、「v1.1.0 をリリースして」）を受け取ることがある。
省略されてもよい（ステップ3で変更内容から提案する）。明示された場合もそのまま鵜呑みにせず、semver 的に妥当かステップ3で確認する。

## ワークフロー

下記を順番に進める。**各ステップを飛ばさない。** 特にステップ5（公開前の最終確認）を省いて公開しないこと。

### 1. 規約とコンテキストの把握

- **`RELEASING.md` を読む**（書式・バージョニング・タグ規則・公開手順の手本。以降はこれに従う）。
- 現在のバージョンを確認: `manifest.json` の `version`。
- 直近のリリース／タグを確認:

  ```sh
  gh release list
  git describe --tags --abbrev=0   # 直近タグ。無ければ「初回リリース」として扱う
  ```

- リポジトリの状態を確認: 現在ブランチ（通常は `main`）、未コミットの変更（`git status`）。想定と違えばユーザに確認する。

### 2. 変更内容の収集

直近タグ以降の変更を集め、リリースノートの**材料**としてユーザに提示する（この時点では確定しない）。

```sh
git log <直近タグ>..HEAD --no-merges --format='%h %s'   # 初回リリースなら範囲指定なしで全件
gh pr list --state merged --base main --limit 30        # マージ済み PR も参考に
```

初回リリースでタグが無い場合は全履歴が対象。件数が多いときは主要な変更に絞ってよいが、**絞ったことは明示する**。

### 3. バージョンの決定と manifest 更新（提案）

集めた変更内容から、semver（`RELEASING.md` の基準）に沿って次バージョンを**提案**する。

- 新機能の追加 → MINOR、後方互換のバグ修正 → PATCH、後方互換性を壊す変更 → MAJOR。
- 引数でバージョンが指定されていても、内容と食い違うと感じたら指摘して確認する。
- ユーザと合意したら **`manifest.json` の `version` を更新**する。

**コミットはユーザの領分。** manifest を更新したら、その差分を**ユーザにコミットしてもらう**よう促し、コミット済みになってから次へ進む。
これは、タグはコミットに対して打たれるため、**バージョン更新を含んだコミットにタグを乗せる**必要があるから（`RELEASING.md` の手順どおり: manifest 更新 → コミット → タグ付け）。未コミットの変更はタグに含まれない点に注意。

### 4. リリースノート本文の作成（対話）★

ステップ2の変更を Keep a Changelog のカテゴリ（`Added` / `Changed` / `Deprecated` / `Removed` / `Fixed` / `Security`）に振り分けた**叩き台**を提示し、
ユーザと対話で文言・粒度・カテゴリ分けを調整して本文を確定する。書式は `RELEASING.md` のテンプレートに従う。

- **該当する変更が無いカテゴリの見出しは省く。**
- コミットメッセージをそのまま貼らず、利用者目線の変更点に言い換える。
- 「妥当そうな叩き台ができた」ことは確定の理由にならない。**本文はユーザの合意をもって確定する。**

### 5. 公開前の最終確認 ★最重要

公開は取り消しの効きにくい外部公開操作。実行前に以下を**まとめて提示し、明示的な承認を得る**:

- タグ名（`vX.Y.Z`）と対象コミット（`git rev-parse --short HEAD`）
- リリースタイトルと本文（確定版）
- 公開（publish）か下書き（`--draft`）か

ユーザが承認するまで `gh release create` を実行しない。

### 6. 公開

確定した本文をファイルに書き出し、`gh` で公開する（本文は `--notes-file` で渡すと引用の崩れが無い）。

```sh
gh release create vX.Y.Z \
  --target "$(git rev-parse HEAD)" \
  --title "vX.Y.Z" \
  --notes-file <本文ファイル>
```

- **`--target` には完全な SHA（またはブランチ名）を渡す。** 短縮 SHA は `HTTP 422: target_commitish is invalid` で失敗する。
- 下書きにする場合は `--draft` を付ける。
- タグ `vX.Y.Z` が既に存在する場合は上書きされない。バージョン重複ならユーザに確認する。

### 7. 事後確認と報告

```sh
gh release view vX.Y.Z
```

で公開結果（タイトル・本文・draft 状態）を確認し、リリースページ URL とともに報告する。
併せて、ユーザ側の残作業（`manifest.json` のバージョン更新コミットがまだなら促す、タグをローカルに取り込むなら `git fetch --tags`）を伝える。

## やらないこと

- ステップ5の最終確認を省いて公開すること。ユーザ承認なしにリリースを公開しない。
- リリースノート本文を対話で固めず、コミットログから独断で確定すること。
- タグ付け・Release 公開・`manifest.json` の version 更新**以外**の git 操作（機能コードのコミット、プッシュ、PR 作成）。これらはユーザの領分。
- リリースの範囲を超えた変更（気づいた点は報告に留める）。
