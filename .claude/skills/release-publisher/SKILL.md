# Release Publisher — リリース PR 作成スキル

「変更を集める → 対話でリリースノートを固める → バージョンを決める → リリース PR を作成する」の流れで
リリースを準備する。PR を `main` にマージすると CI が GitHub Release を自動公開する。

このスキルの肝は 2 つ:

1. **本文はユーザとの対話で確定する。** コミット履歴はあくまで叩き台の材料であり、勝手に確定しない。
2. **公開は PR マージで行われる。** スキルは PR を作るところまでを担い、マージ判断はユーザに委ねる。

書式・バージョニング・タグ命名などの規約は **`RELEASING.md` を唯一の出典とする**（ここに重複して書かず、必ず読んで従う）。

## 入力

公開したいバージョン（例: `release-publisher 1.2.0`、「v1.2.0 をリリースして」）を受け取ることがある。
省略されてもよい（ステップ3で変更内容から提案する）。明示された場合もそのまま鵜呑みにせず、semver 的に妥当かステップ3で確認する。

## ワークフロー

下記を順番に進める。**各ステップを飛ばさない。**

### 1. 規約とコンテキストの把握

- **`RELEASING.md` を読む**（書式・バージョニング・タグ規則・公開手順の手本。以降はこれに従う）。
- 現在のバージョンを確認: `manifest.json` の `version`。
- 直近のリリース／タグを確認:

  ```sh
  gh release list
  git describe --tags --abbrev=0   # 直近タグ。無ければ「初回リリース」として扱う
  ```

- リポジトリの状態を確認: 現在ブランチ、未コミットの変更（`git status`）。`main` 以外にいる場合はユーザに確認する。

### 2. 変更内容の収集

直近タグ以降の変更を `main` ブランチから集め、リリースノートの**材料**としてユーザに提示する（この時点では確定しない）。

```sh
git log <直近タグ>..main --no-merges --format='%h %s'   # 初回リリースなら範囲指定なしで全件
gh pr list --state merged --base main --limit 30         # マージ済み PR も参考に
```

初回リリースでタグが無い場合は全履歴が対象。件数が多いときは主要な変更に絞ってよいが、**絞ったことは明示する**。

### 3. バージョンの決定

集めた変更内容から、semver（`RELEASING.md` の基準）に沿って次バージョンを**提案**する。

- 新機能の追加 → MINOR、後方互換のバグ修正 → PATCH、後方互換性を壊す変更 → MAJOR。
- 引数でバージョンが指定されていても、内容と食い違うと感じたら指摘して確認する。
- ユーザと合意してから次へ進む。

### 4. リリースノート本文の作成（対話）★

ステップ2の変更を Keep a Changelog のカテゴリ（`Added` / `Changed` / `Deprecated` / `Removed` / `Fixed` / `Security`）に振り分けた**叩き台**を提示し、
ユーザと対話で文言・粒度・カテゴリ分けを調整して本文を確定する。書式は `RELEASING.md` のテンプレートに従う。

- **該当する変更が無いカテゴリの見出しは省く。**
- コミットメッセージをそのまま貼らず、利用者目線の変更点に言い換える。
- 「妥当そうな叩き台ができた」ことは確定の理由にならない。**本文はユーザの合意をもって確定する。**

### 5. リリース PR の作成

バージョンとリリースノートが確定したら、リリース用ブランチを作り PR を作成する。

1. `main` から `release/vX.Y.Z` ブランチを作成する。
2. `manifest.json` の `version` を更新する。
3. `RELEASE_NOTES.md` に確定したリリースノート本文を書き出す。
4. 変更をコミットし、PR を作成する:

   ```sh
   git checkout -b release/vX.Y.Z main
   # manifest.json と RELEASE_NOTES.md を更新
   git add manifest.json RELEASE_NOTES.md
   git commit -m "release: vX.Y.Z"
   git push -u origin release/vX.Y.Z
   gh pr create --title "release: vX.Y.Z" --body "<リリースノート本文>"
   ```

5. PR の URL をユーザに報告し、レビュー・マージを促す。

**PR をマージすると CI が自動で GitHub Release を公開する**旨を伝える（CI は `manifest.json` の変更を検知し、バージョンに対応するタグが無ければ `RELEASE_NOTES.md` の内容でリリースを作成する）。

### 6. 事後（マージ後）

スキルの範囲はステップ5まで。マージ後にユーザから確認を求められた場合は:

```sh
gh release view vX.Y.Z
```

で公開結果（タイトル・本文・draft 状態）を確認し報告する。

## やらないこと

- リリースノート本文を対話で固めず、コミットログから独断で確定すること。
- `gh release create` を直接実行すること（CI の役割）。
- リリース PR 以外の git 操作（機能コードのコミット、別 PR の作成）。
- リリースの範囲を超えた変更（気づいた点は報告に留める）。
