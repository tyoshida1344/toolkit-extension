# Review Perf — パフォーマンスレビュー

変更差分を「コード量を減らせないか」「処理を軽くできないか」の2軸で確認する。

## 対象の特定

1. 変更ファイルを取得する:

```sh
git diff main --name-only
```

2. 変更ファイルの全体を読む（diff だけでなく前後の文脈が必要）
3. 変更ファイルを参照・利用している関連ファイルも特定して読む（`Toolkit.*` API の利用元、同じ CSS クラスを使うファイルなど）

## 確認観点

以下の観点で変更箇所とその周辺を確認する。全てを機械的にチェックするのではなく、**実際に効果がある指摘** に絞る。

### コード量の削減

- 既存の `Toolkit.*` ヘルパー（`copyButton`, `bindState`, `modal`, `getTabsById`, `tryRegex`, `iconButton`, `showToast` 等）で置き換えられる独自実装がないか
- 重複するロジックを関数に抽出できないか
- 不要な変数・関数・条件分岐がないか
- 同じ DOM 要素への複数回アクセスを変数にキャッシュできないか
- テンプレートリテラルで簡潔にできる文字列結合がないか

### 処理の軽量化

- `querySelectorAll` + ループを `querySelector` 1 回で済ませられないか
- 高頻度イベント（input, scroll, resize）にデバウンス / スロットルが必要でないか
- `chrome.storage` への個別アクセスを一括取得にまとめられないか
- ループ内の不変な計算をループ外に出せないか
- 文字列の `+=` 連結を `Array` + `join` に置き換えられないか
- 不要な DOM 再描画（innerHTML の全書き換えなど）を部分更新にできないか

## 出力

指摘事項を以下の形式で報告する:

```
### パフォーマンスレビュー結果

**コード量削減**
- [ ] `ファイル名:行番号` — 指摘内容（改善案）

**処理軽量化**
- [ ] `ファイル名:行番号` — 指摘内容（改善案）

指摘なしの場合: 「パフォーマンス上の問題は見つかりませんでした」
```

修正が求められている場合（issue-handler 経由）は、報告後に指摘箇所を修正する。
