# Issue Handler — イシュー対応スキル

GitHub イシューを「ブランチ準備 → 読む → 不明点を潰す → 仕様を確定して記録する → 実装する → 検証する → セルフレビュー → PR 作成」の流れで対応する。
このスキルの価値は **実装を急がず、イシューに潜む曖昧さを実装前に全て解消すること** にある。
イシューは多くの場合「やりたいこと」を口語で書いただけで、判断が必要な仕様がいくつも未確定のまま残っている。
そこを勝手に決めて作ると手戻りになるので、必ずユーザに確認してから手を動かす。

## 入力

イシュー番号を受け取る。引数（例: `issue-handler 8`）でも、文中（「#8 を対応して」）でもよい。
番号が読み取れないときは、どのイシューを対応するか先にユーザへ確認する。

## ワークフロー

下記を順番に進める。**各ステップを飛ばさない。** 特にステップ4（不明点の確認）を省いて実装に入らないこと。

### 1. ブランチ準備

最新の main ブランチから作業用ブランチを作成する。

まず `git status` で現在の状態を確認する。以下のいずれかに該当する場合は**警告してユーザに確認**してから進める:
- 未コミットの変更（staged / unstaged / untracked）がある
- main 以外のブランチにいる

ユーザが続行を許可したら:

```sh
git checkout main
git pull origin main
git checkout -b issues/<番号>
```

ブランチ名は `issues/<イシュー番号>` で統一する（例: `issues/45`）。

### 2. イシューの取得と理解

まずイシュー本文と既存コメントを取得する。コメントには過去の議論や追加の決定が入っていることがあるので必ず読む。

```sh
gh issue view <番号> --comments
```

タイトル・本文・コメントから「何を新規に作るのか / 何を直すのか」「背景・動機」を把握する。
取得に失敗する（番号が存在しない等）場合は、その旨をユーザに伝えて番号を確認する。

### 3. プロジェクト規約の把握

実装が規約から外れないよう、先に書き方の手本を読む。

- `CLAUDE.md`（あれば最優先。アーキテクチャ・命名・配置・検証方法の指針が書かれている）
- `README`、および **同種の既存ファイル**（新機能なら近い既存機能のソースを 1〜2 本読み、構造を踏襲する）

ここで掴んだパターン（共通ヘルパー、ファイル配置、登録の作法など）に実装を合わせる。

### 4. 不明点の洗い出しと確認 ★最重要

イシューの記述を、**実装するなら必ず答えが要る粒度** まで分解し、決まっていない点を全て挙げる。
典型的に曖昧になりやすいのは:

- 機能の範囲（どこまでやる / やらない、optional な要望の優先度）
- 具体的な数値・上限・初期値（「上限を設けたい」→ 何個？）
- 操作・入力方法（キーボード対応の要否、ボタン UI の有無 など）
- UI 上の配置・名称・表示文言、既存機能との関係
- 参照されている外部の固有名詞（「○○と同じようなUIに」→ それは何を指すのか）
- 異常系・エッジケースの扱い

挙げた不明点は **AskUserQuestion でまとめて提示** し、ユーザの回答で全て解消してから次へ進む。
一問一答で往復を増やすより、関連する論点をまとめて聞く方がユーザの負担が小さい。
本当に解釈の余地がない点まで確認する必要はないが、迷ったら聞く側に倒す。

**「妥当なデフォルトを思いつく」ことは、確認を省く理由にはならない。**
ユーザはこのスキルを通じて *自分で仕様を決めたい* のであって、もっともらしい既定値で代用されたいわけではない。
仕様を左右する判断（範囲・数値・挙動・UI の形など）を勝手に確定すると、たとえ動くものができても意図と違えば作り直しになる。
だからこそ、決めるのはユーザで、こちらの役割は「決めるべき論点を漏れなく差し出すこと」だと意識する。
回答がすぐ得られない状況でも、その論点を勝手に確定して実装で先回りするのではなく、確認待ちであることを明示して止まる方がよい。

### 5. 仕様の確定とイシューへの記録

ユーザの回答を反映して最終仕様をまとめ、**実装に入る前に** イシューへコメントとして残す。
これは「何を作ると合意したか」の記録になり、後から実装の意図を追えるようにするため。

```sh
gh issue comment <番号> --body "$(cat <<'EOF'
## 確定仕様

### 概要
<1〜2行で何を作る/直すか>

### 詳細仕様
- <確定した仕様を箇条書き>

### 確認した不明点と決定
- Q: <聞いたこと> → A: <決まったこと>

### 実装方針
- <変更するファイル / 追加するモジュール / 使う既存パターン>
EOF
)"
```

コメント本文はイシューの内容に合わせて過不足なく。決定が無い節は省いてよい。

### 6. 実装

**ステップ4の確認が済むまでファイルは編集しない。** 実装への着手はユーザの回答を得てから。
確定仕様とステップ3で掴んだ規約に沿って実装する。既存の共通ヘルパー・パターンを使い、独自実装を増やさない。
バグ修正イシューでも流れは同じ（原因を特定 → 直し方を仕様として確定 → 修正）。

### 7. 検証

プロジェクトの検証方法に従って、完了とみなす前に最低限の確認を行う。
`CLAUDE.md` に検証コマンドがあればそれを使う（例: このリポジトリなら編集したファイルへの `node --check`）。
検証コマンドが無いプロジェクトでも、構文・明らかな破綻が無いかは確かめる。

### 8. セルフレビューと修正

実装・検証が完了したら、3つの観点でセルフレビューを行い、指摘事項を修正する。

1. 以下のレビューを順に実行し、各レビューの指摘事項を **修正** する:
   - `/review-perf` — コード量削減・処理軽量化の観点
   - `/review-regression` — デグレ・既存機能への影響確認
   - `/review-ux` — UX 面の改善点確認
2. 修正が入った場合は、再度構文検証（`node --check` など）を実施する
3. 修正しきれなかった指摘事項があれば、完了報告に含める

### 9. コミットと PR 作成

セルフレビューが完了したら、変更をコミットして PR を作成する。

1. コミット:

```sh
git add <変更したファイル>
git commit -m "prefix: #<番号> 説明"
```

- prefix はイシューの内容に応じて判別する（`feat` / `fix` / `refactor` / `chore` 等）

2. PR 作成: `.github/PULL_REQUEST_TEMPLATE.md` を読み、各セクションを埋めた内容を `--body` に渡す。

```sh
git push -u origin issues/<番号>
gh pr create --title "prefix: #<番号> 説明" --body "<テンプレートを埋めた内容>"
```

- Summary: 変更内容を利用者・レビュアー視点で箇条書き
- Test plan: 手動で確認すべき項目をチェックボックスで列挙し、実施済みならチェックを入れる
- `Closes #<番号>` にイシュー番号を記入

### 10. 完了報告

最後に以下を簡潔に報告する:

- 確定した仕様の要約（とイシューへコメントした旨）
- 変更したファイルと、何をしたか
- 検証結果
- セルフレビューの結果（残った指摘事項があれば）
- PR の URL

## やらないこと

- ステップ4を飛ばして、曖昧なまま想定で実装すること
- イシューの要望を超えた余計な変更（気づいた改善点は報告に留め、別途相談する）
